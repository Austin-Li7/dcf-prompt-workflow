## v5.4 当前整体状态
如果一句话概括：

**v5.4 已经不是单纯“会写分析”的 prompt，而是一套结构已经成型的 validation working system。**

它现在已经具备：
- 按步骤推进的 7-step backbone
- claims-first 的结构化输出方式
- canonical name registry
- disclosure-bounded extraction
- low-disclosure fallback
- arithmetic trace
- driver eligibility
- final pipeline audit

这意味着它已经可以输出一条相对完整的 validation 链路，而不是停留在单点分析。

从实际效果看，v5.4 的优势主要在三个方面：
1. **结构化程度更高**：每一步已经不只是 narrative，而是开始输出 claims、JSON、registry、tables、assumption blocks 等可继续传递的结果。
2. **防止乱补数据的能力更强**：尤其在 Step 2 和 Step 5 中，对披露边界、缺失数据和不应建模的对象控制得更严格。
3. **forecast 治理能力明显提升**：不仅给预测结果，还开始给出 basis、driver quality、weak inference exposure 和 arithmetic trace。

整体上可以理解为：
- 它已经具备较强的结构纪律和一定的真实可用性；
- 它已经能支持完整 validation 流程的试跑；
- 后续提升重点，更多是**优化传递效率、降低执行负担、让输出更短更稳**，而不是继续无限加规则。

## 原始 v5.4 的核心变化
根据原文件头部摘要，v5.4 主要新增或强化了这些点：
1. Step 5 Arithmetic Trace：assumption registry 必须包含数值推导链。
2. Step 5 Segment-Only Fallback：低披露公司自动切到 segment-level forecast。
3. Step 3 拆成 3A + 3B：competitor pairing 和 Porter 分开，降低指令密度。
4. Light Mode 实质简化：Step 3/4/5 各自有独立简化版 generation prompt。
5. Step 2 Restatement History Depth：CAGR lookback 限于 restated period。
6. Step 4.5 Asset-Light Threshold：CapEx/Revenue < 8% 时 ceiling 可标记为不适用。
7. Steps 2–7 增加 name consistency 指令。
8. Step 4 Differentiation 默认降到 [WEAK_INFERENCE]，Flywheel 必须引用 Step 2。
9. Step 7 Light Mode 合并 Checkpoint A + C。
10. 因果纪律指令去重：主要放在 header 统一定义。

## 每一步输出什么，以及是否能传到下一步

### Step 1 — Business Architecture
**主要输出：**
- structured claims
- architecture JSON
- excluded items
- canonical name registry

**是否能传到下一步：**
- 可以，而且传递效果很好。
- Step 2–Step 7 都会用到 Step 1 定义的 canonical names 和业务结构。

**当前效果：**
- 这是 v5.4 里质量较高的一步。
- 它不仅解释公司业务，也真正定义了后续允许分析哪些 segment / product，以及这些对象应该怎么命名。

### Step 2 — Historical Financial Data
**主要输出：**
- disclosure inventory
- structured data claims
- 历史财务表 / CSV
- rollup validation
- anomaly flags

**是否能传到下一步：**
- 可以，而且是整条链路最关键的输入之一。
- Step 4、Step 4.5、Step 5、Step 7 都会实际消费 Step 2 的结果。

**当前效果：**
- 这是 v5.4 最有实际价值的步骤之一。
- 它最大的作用是把“哪些数字能提、哪些不能提、哪些只能用 growth rate”这件事结构化了，后续 forecast 约束明显更稳。

### Step 3 — Competitive Landscape
**主要输出：**
- Step 3A: competitor pairing
- rivalry intensity
- source tier / provenance
- Step 3B: Porter’s Five Forces（Full Mode）

**是否能传到下一步：**
- 可以传，但真正被消费的主要是 competitor identity、rivalry level 和少量 competitive constraints。

**当前效果：**
- 这一步的结构是成立的，尤其 3A 很有用。
- 它帮助后续 forecast 不至于完全脱离竞争约束。
- 后续还可以继续提升的方向，是让这一步输出更聚焦于真正会被 Step 4/5 消费的内容。

### Step 4 — Synergy & Flywheel
**主要输出：**
- Full Mode: Integration / Differentiation / Financial Causality 三套 verdict
- Light Mode: Integration / Causality 两套 verdict
- synergy classification
- Step 5 driver eligibility mapping

**是否能传到下一步：**
- 可以，而且对 Step 5 很重要。
- Step 5 主要消费的是：这个 synergy 能不能 drive、能 drive 到什么程度、是否需要 cap。

**当前效果：**
- 这一步的治理思想是成熟的。
- 它最大的价值，不是写出一段 synergy narrative，而是把“哪些故事只是 context，哪些能真正进入 forecast”区分开。
- 后续还可以继续优化的方向，是进一步压缩冗余表述，让分类结果和 driver eligibility 更突出。

### Step 4.5 — Capital Allocation / Feasibility
**主要输出：**
- capital claims
- capital metrics table
- feasibility checkpoints
- revenue ceiling JSON
- asset-light exemption tag

**是否能传到下一步：**
- 可以，而且传递得比较干净。
- Step 5 会直接消费 feasibility、ceiling 或 asset-light exemption。

**当前效果：**
- 这是 v5.4 里比较稳定的一步。
- 它让 forecast 不只是基于“增长逻辑”，还开始受到 reinvestment feasibility 的约束。

### Step 5/6 — Forecasting
**主要输出：**
- assumption registry
- arithmetic trace
- forecast tables
- consolidated master forecast
- sensitivity tables
- weak-inference sensitivity
- forecast confidence summary

**是否能传到下一步：**
- 可以，它本身就是整条链路的数值核心。
- 前面 Step 1–4.5 的结果会在这里被集中消费，后面由 Step 7 和 Pipeline Audit 做最终检查与收口。

**当前效果：**
- 这是 v5.4 最核心、也最有价值的一步。
- 特别是 arithmetic trace、segment-only fallback、basis_claim_id、data resolution constraint，这些都显著提升了 forecast 的可解释性和纪律性。
- 后续还可以提升的方向，是进一步提高执行效率和输出压缩度，让这一步在复杂公司场景下更稳定。

### Step 7 — WACC
**主要输出：**
- parameter claims
- staleness / source information
- WACC input set
- beta mapping / segment weighting
- reasonableness and completeness checks

**是否能传到下一步：**
- 可以，主要服务于最终 valuation 和 final audit。

**当前效果：**
- 这一步的纪律性比较强。
- 它的主要价值在于把 WACC 输入参数从“黑箱常数”变成“可追溯参数集”。

### Pipeline Audit
**主要输出：**
- final audit JSON
- structural issues
- numerical issues
- evidence laundering check
- weak-inference exposure
- final recommendation

**是否能传到下一步：**
- 可以，主要作为最终终审和是否 proceed 的判断依据。

**当前效果：**
- 这一步让整套系统具备了最后的 cross-step consistency 检查能力。
- 它的价值在于做收口，确认前面各步是否真的闭合，而不只是“每一步看起来都像对的”。

## 哪些步骤传递效果最好
从实际使用角度看，当前 v5.4 里传递最顺的步骤主要是：
- **Step 1**
- **Step 2**
- **Step 4.5**
- **Step 7**

这些步骤的输出对象比较清晰，后面步骤也能比较直接地消费它们。

而 **Step 3** 和 **Step 4** 的逻辑价值也很高，只是其中一部分内容会比真正被下游消费的部分更丰富一些。它们的提升方向不是推翻，而是继续收束、提高传递效率。

## 整体质量和可用性
如果从“真实工作里能不能用”这个角度看，当前 v5.4 可以理解为：

**已经是一个有清晰结构、能跑出完整 validation 结果的 working system。**

它现在已经做到了：
- 单步输出大多有明确结构；
- 多数关键步骤已经可以把结果传到下一步；
- 披露边界、forecast basis、数值闭环这几件事，比早期版本明显更稳；
- 对复杂 validation 工作，已经有较强的支撑能力。

如果从阶段上看，v5.4 更适合被理解为：
- 一个已经成型的版本；
- 一个具备较强纪律性的版本；
- 一个后续重点应该放在“优化传递效率、简化执行方式、提高运行稳定性”的版本。

## 推荐使用方式
1. 先读 `01_STEP_1...`，建立 architecture JSON 和 canonical name registry。
2. 再跑 `02_STEP_2...`，得到 disclosure inventory、历史数据表、rollup check。
3. 再进入 `03_STEP_3...`，Light Mode 只用 3A，Full Mode 用 3A + 3B。
4. `04_STEP_4...` 根据模式选择 Full 或 Light 版本。
5. `05_STEP_4_5...` 给 Step 5 提供 feasibility / ceiling 约束。
6. `06_STEP_5_6...` 是预测主干，Light Mode 用 segment-level annual forecast。
7. `07_STEP_7...` 处理 WACC 参数级校验。
8. 最后跑 `08_PIPELINE_AUDIT...` 做总审计。

## 建议
- 如果是第一次跑，建议优先从 **Light Mode** 开始。
- 如果公司披露较弱，优先接受 v5.4 的 **segment-level forecast** 逻辑，而不要强行做 product-level 细分。
- 如果后续继续迭代，这套系统最值得保留的部分是：
  - claims-first
  - canonical names
  - disclosure inventory
  - segment-only fallback
  - arithmetic trace
  - driver eligibility
  - final pipeline audit

这些是 v5.4 当前最核心、也最有实际价值的部分。
